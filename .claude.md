# Guide de développement OnlyRoll

Ce fichier sert de référence pour tous les développements du projet OnlyRoll, une plateforme de jeu de rôle en ligne avec gestion de parties, maps virtuelles, tokens et chat temps réel.

---

## Stack technique

### Backend
- **Framework :** Symfony 6
- **Langage :** PHP 8.3
- **Base de données :** MySQL
- **ORM :** Doctrine
- **API :** REST (JSON)
- **Authentification :** JWT (LexikJWTAuthenticationBundle)
- **WebSockets / temps réel :** Mercure
- **Tests :** PHPUnit
- **Norme de code :** PSR-12
- **Analyse statique :** PHPStan

### Frontend
- **Framework :** Vue.js 3 (Composition API)
- **Langage :** TypeScript
- **Gestion d'état :** Pinia
- **Routing :** Vue Router
- **Requêtes HTTP :** Axios
- **UI :** Tailwind CSS + composants personnalisés
- **Tests :** Vitest + Playwright (E2E)

### Environnement
- **Conteneurisation :** Docker Compose (à venir plus tard)
- **Versioning :** Git (GitLab)
- **CI/CD :** GitLab CI

## Architecture du projet

### Backend (`backend/`)

```
src/
├── Controller/          # Controllers REST API (légers, délèguent aux services)
├── Service/             # Logique métier (GameService, ChatService, MapService, etc.)
├── Entity/              # Entités Doctrine (User, Game, GameMap, GameToken, etc.)
├── Repository/          # Accès aux données uniquement
├── DTO/                 # Data Transfer Objects (validation, sérialisation)
├── Enum/                # Énumérations (GameStatus, PlayerRole, PlayerStatus)
├── Exception/           # Exceptions métier personnalisées
├── EventListener/       # Écouteurs d'événements
├── EventSubscriber/     # Souscripteurs d'événements
└── Security/            # Authentification JWT custom

tests/
├── Unit/                # Tests unitaires (Entity, Service)
└── Functional/          # Tests fonctionnels (Controller)
```

**Services principaux :**
- `GameService` : gestion des parties
- `ChatService` : messages de chat
- `MapService` : gestion des maps
- `TokenService` : gestion des tokens/pions
- `MercurePublisher` : publication temps réel
- `FileUploader` : upload de fichiers (avatars, maps, tokens)
- `DtoValidatorService` : validation des DTOs

### Frontend (`frontend/`)

```
src/
├── components/          # Composants Vue réutilisables
│   ├── auth/           # LoginForm, RegisterForm
│   ├── common/         # FeatureCard, UserProfileBadge
│   ├── game/           # ChatPanel, GameMap, DiceRoller, etc.
│   └── dashboard/      # DashboardCard
├── views/              # Pages complètes
│   ├── auth/           # LoginView, RegisterView
│   ├── dashboard/      # DashboardView
│   └── games/          # GameListView, GamePlayView
├── stores/             # Stores Pinia (auth, game, chat, map)
├── services/           # Services API (axios clients)
├── composables/        # Logique réutilisable (useAuth, useMercure)
├── router/             # Configuration Vue Router
├── types/              # Types TypeScript
└── utils/              # Fonctions utilitaires
```

## Règles et conventions

### Code général
- Respecter les normes **PSR-12** pour le PHP
- Commenter les fonctions complexes avec des **docblocks PHPDoc**
- Toujours **sanitizer** les données utilisateurs avant insertion
- Ne jamais exposer d'informations sensibles dans les réponses API
- Privilégier la **composition** plutôt que l'héritage
- Respecter le principe **SOLID**

### Gestion de la structure du projet
- **IMPORTANT** : Chaque fois qu'un fichier est créé, déplacé ou supprimé, mettre à jour le fichier `structure.md` à la racine du projet
- Le fichier `structure.md` doit refléter fidèlement l'arborescence complète du projet
- Respecter le format markdown existant avec l'indentation appropriée
- Cette règle s'applique à **tous** les fichiers (backend, frontend, configuration, documentation)

### Backend (Symfony)

#### Architecture
- Les **Repository** gèrent uniquement la **logique d'accès aux données** (findBy, custom queries)
- Les **Controller** restent légers et déléguent la logique aux **services**
- Les **Services** contiennent la logique métier
- Les **DTOs** servent à la validation et la sérialisation des données
- Les **Entities** ne doivent contenir que des getters/setters et relations Doctrine

#### Sécurité
- Utiliser le trait `Securite` et sa méthode `sanitize()` pour toutes les entrées utilisateurs
- Valider systématiquement les DTOs avec le `DtoValidatorService`
- Vérifier les permissions d'accès dans les services (ex: seul le MJ peut modifier la partie)
- Ne jamais exposer les mots de passe ou tokens JWT en clair

#### Bonnes pratiques
- Utiliser les **Enums PHP 8.1+** pour les états (GameStatus, PlayerRole, etc.)
- Créer des **exceptions métier** personnalisées (GameFullException, AccessDeniedException)
- Typer strictement les retours de méthodes et paramètres
- Utiliser l'injection de dépendances
- Ne jamais faire de `doctrine:schema:update`, toujours utiliser les **migrations**

#### Tests
- Les **tests unitaires** doivent être simples, sans dépendance à la base réelle
- Utiliser des **mocks** pour les dépendances externes
- Tester les cas nominaux ET les cas d'erreur
- Minimum **80% de couverture** pour les services critiques

### Frontend (Vue.js)

#### Architecture
- Préférer des composants **modulaires et réutilisables**
- Utiliser la **Composition API** systématiquement
- Extraire la logique complexe dans des **composables**
- Gérer l'état global avec **Pinia**
- Isoler les appels API dans des **services dédiés**

#### TypeScript
- **Typage strict** activé (`"strict": true`)
- Définir des **interfaces** pour tous les objets métier
- Éviter `any`, préférer `unknown` si nécessaire
- Typer les props, emits et retours de composables

#### Nommage
- **PascalCase** pour les composants Vue (`LoginForm.vue`, `GameCard.vue`)
- **camelCase** pour les variables et fonctions
- **kebab-case** pour les noms de fichiers non-composants
- Noms de variables **explicites et cohérents** (éviter `data`, `item`, préférer `game`, `player`)

#### Tailwind CSS
- Utiliser les classes utilitaires Tailwind
- Créer des composants réutilisables pour les patterns répétitifs
- Éviter le CSS custom sauf nécessité absolue
- Respecter le design system (couleurs, espacements, etc.)

#### Tests
- Tester les composants avec **Vitest**
- Tester les stores Pinia
- Tests E2E avec **Playwright** pour les workflows critiques
- Mocker les appels API

## Sécurité

### Général
- **CORS** configuré strictement (nelmio_cors)
- **HTTPS** en production
- **Sanitization** de toutes les entrées utilisateurs
- **Validation** systématique des données (DTOs + Symfony Validator)

### Authentification
- Tokens JWT stockés en **httpOnly cookies** (non accessible en JS)
- Refresh token pour renouveler l'accès
- Expiration des tokens (15min pour access, 7j pour refresh)
- Logout côté serveur (blacklist des tokens)

### Uploads
- Vérifier le **type MIME** des fichiers
- Limiter la **taille des uploads**
- Stocker les fichiers en dehors du document root si possible
- Renommer les fichiers uploadés (éviter les injections)

```php
namespace App\Service;

use App\Entity\Game;

/**
 * Service de gestion des parties.

final class GameService

    public function __construct(
        private readonly GameRepository $gameRepository,

    }

    /**
     * Crée une nouvelle partie.
     *
     * @param CreateGameDTO $dto Données de la partie
     * @return Game La partie créée
     * @throws GameException Si erreur de création
     */
    public function createGame(CreateGameDTO $dto): Game
    {
        // Logique métier ici
    }
}
```

**Points clés :**
- `declare(strict_types=1)` en début de fichier
- Classes `final` par défaut (sauf si héritage nécessaire)
- `readonly` pour les dépendances injectées
- Docblocks pour les méthodes publiques
- Typage strict des paramètres et retours

### Frontend (TypeScript + Vue)

```vue
<script setup lang="ts">
import { ref, computed } from 'vue'
import type { Game } from '@/types/game'

interface Props {
  game: Game
  showActions?: boolean
}

interface Emits {
  (e: 'join', gameId: number): void
  (e: 'delete', gameId: number): void
}

const props = withDefaults(defineProps<Props>(), {
  showActions: true
})

const emit = defineEmits<Emits>()

const isLoading = ref(false)

const playerCount = computed(() => props.game.players.length)

const handleJoin = async (): Promise<void> => {
  isLoading.value = true
  try {
    emit('join', props.game.id)
  } finally {
    isLoading.value = false
  }
}
</script>

<template>
  <div class="bg-white rounded-lg shadow-md p-6">
    <h3 class="text-xl font-bold mb-2">{{ game.name }}</h3>
    <p class="text-gray-600">{{ playerCount }} joueurs</p>
    <button
      v-if="showActions"
      @click="handleJoin"
      :disabled="isLoading"
      class="mt-4 px-4 py-2 bg-blue-600 text-white rounded hover:bg-blue-700 disabled:opacity-50"
    >
      Rejoindre
    </button>
  </div>
</template>
```

**Points clés :**
- `<script setup lang="ts">` systématiquement
- Interfaces pour Props et Emits
- Types importés depuis `@/types/`
- Nommage explicite des handlers (`handleJoin` et non `join`)
- Classes Tailwind pour le styling

### Patterns spécifiques au projet

#### Gestion des parties (Game)
- Une partie a un statut (`GameStatus`: WAITING, STARTED, FINISHED)
- Seul le **MJ (Game Master)** peut modifier la partie
- Les joueurs peuvent avoir différents rôles (`PlayerRole`: GAME_MASTER, PLAYER)
- Les joueurs ont un statut (`PlayerStatus`: ACTIVE, INACTIVE, BANNED)

#### Maps et Tokens
- Une partie peut avoir plusieurs **maps**
- Chaque map contient des **tokens** (pions/personnages)
- Les positions sont stockées en coordonnées (x, y)
- Uploads de fichiers pour les images de maps et tokens

#### Chat temps réel
- Utilise **Mercure** pour la diffusion temps réel
- Les messages sont liés à une partie
- Format : `{user, message, timestamp}`

#### Architecture DTO
- Tous les inputs API doivent passer par des **DTOs**
- Validation avec Symfony Validator (`@Assert`)
- Sanitization avec le trait `Securite`

## Exemple de tâche attendue

**Demande utilisateur :**
> "Crée-moi un service Symfony pour envoyer un email de validation d'inscription via Mailer, et un composant Vue correspondant à la page de confirmation."

**Réponse attendue de Claude :**

1. **Backend** : Créer `EmailService.php` dans `backend/src/Service/`
```php
<?php

declare(strict_types=1);

namespace App\Service;

use Symfony\Component\Mailer\MailerInterface;
use Symfony\Component\Mime\Email;

final class EmailService
{
    public function __construct(
        private readonly MailerInterface $mailer,
    ) {
    }

    public function sendRegistrationConfirmation(string $to, string $username): void
    {
        $email = (new Email())
            ->from('noreply@onlyroll.com')
            ->to($to)
            ->subject('Bienvenue sur OnlyRoll')
            ->html("<p>Bonjour {$username}, votre inscription est confirmée !</p>");

        $this->mailer->send($email);
    }
}
```

2. **Configuration** : Enregistrer le service dans `backend/config/services.yaml` (si nécessaire, souvent auto-configuré)

3. **Frontend** : Créer `ConfirmationView.vue` dans `frontend/src/views/auth/`
```vue
<script setup lang="ts">
import { onMounted } from 'vue'
import { useRouter } from 'vue-router'

const router = useRouter()

onMounted(() => {
  setTimeout(() => {
    router.push({ name: 'login' })
  }, 3000)
})
</script>

<template>
  <div class="flex items-center justify-center min-h-screen bg-gray-100">
    <div class="bg-white p-8 rounded-lg shadow-md text-center">
      <h1 class="text-2xl font-bold text-green-600 mb-4">Inscription réussie !</h1>
      <p class="text-gray-600">Un email de confirmation vous a été envoyé.</p>
      <p class="text-sm text-gray-500 mt-4">Redirection automatique dans 3 secondes...</p>
    </div>
  </div>
</template>
```

4. **Intégration** :
- Appeler `EmailService->sendRegistrationConfirmation()` dans `AuthController` après création du user
- Ajouter la route `/confirmation` dans `frontend/src/router/index.ts`
- Rediriger vers `/confirmation` après inscription réussie

5. **Tests** :
- Créer `EmailServiceTest.php` pour tester l'envoi d'email (avec mock du Mailer)
- Créer `ConfirmationView.spec.ts` pour tester le composant Vue
